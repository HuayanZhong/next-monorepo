## Loop Engineering 规则

当用户要求以 Loop 模式执行任务时，按以下流程运作：

### 智能触发

当任务同时满足以下两个条件时，自动进入 Loop 模式：

1. **有明确的完成状态** — 能通过命令、测试、类型检查等方式客观判断"做完了没"
2. **需要多轮迭代** — 单次执行大概率无法一步到位（批量修改、渐进式修复、覆盖率提升等）

满足时直接进入 Loop，不需要用户额外声明。

不进入 Loop 的情况：

- 单次操作就能完成的任务（如"加个函数"、"改个配置"）
- 无法客观验证的开放性任务（如"优化代码风格"）—— 先和用户确认验收标准

### 执行流程

1. **确认目标** — 和用户确认可程序化验证的验收标准（如 `tsc --noEmit` 零报错、`npm test` 全绿、lint 零警告）
2. **确认边界** — 限定改动范围（哪些文件/目录可以改）
3. **确认上限** — 最大迭代次数（默认 5 轮，用户可调）
4. **进入循环**：
   - 执行修改（调用相关 Skill）
   - 运行验收命令
   - 通过 → 结束，汇报结果
   - 未通过 → 分析错误，带着反馈进入下一轮
   - 达到上限 → 停止，汇报当前状态和卡点

### 可用 Skill 组合

| 场景            | 调用的 Skill              | 评估器                   |
| --------------- | ------------------------- | ------------------------ |
| 类型错误修复    | `Code`                    | `npx tsc --noEmit`       |
| 测试补全        | `test-driven-development` | `npm test -- --coverage` |
| Lint 修复       | `Code`                    | `npx eslint . --fix`     |
| 运行时 Bug 调试 | `TRAE-debugger`           | 测试通过                 |
| 安全漏洞修复    | `security-auditor`        | 扫描零告警               |
| 代码简化        | `code-simplifier`         | `tsc` + `test` 通过      |

### 安全边界

- 每轮只改限定范围内的文件
- 不碰 `.env`、密钥、配置文件
- 每轮跑验收命令，不累积未验证的改动
- 达到上限必须停止，不能无限循环

### 汇报格式

Loop 结束后输出：

- 总共跑了几轮
- 改了什么文件
- 验收命令的最终结果
- 如果未达标，卡在哪里
